Resumen ejecutivo

Este manual explica cómo operar el proyecto infra, la capa de orquestación del ecosistema Omnisalud Digital. infra centraliza el despliegue de cuatro aplicaciones web (Portal Corporativo, Módulo Asistencial, Módulo Comercial y Control de Costos) mediante Docker Compose y un proxy reverso Traefik. Está dirigido a administradores de sistemas y personal de DevOps responsable de desplegar, monitorear y mantener la plataforma en servidores Linux. Todo el ecosistema se levanta con un solo comando desde el directorio infra/.

1. Introducción

1.1 ¿Qué es infra?

infra es el proyecto de orquestación que unifica el despliegue de todo el ecosistema Omnisalud Digital. Contiene el archivo docker-compose.yml que define y conecta los seis servicios del sistema, la configuración del proxy reverso Traefik, y las variables de entorno compartidas entre todas las aplicaciones.

1.3 ¿Qué problemas resuelve?

  • Unifica el despliegue de cuatro aplicaciones independientes en un solo comando.
  • Centraliza la configuración de red, dominios, HTTPS y variables de entorno.
  • Provee ruteo automático por ruta (/app1, /app2, /app3) sin exponer puertos individuales.
  • Facilita la renovación automática de certificados SSL vía Let's Encrypt.
  • Permite reconstruir una sola aplicación sin afectar a las demás.

4.3 SSL y Let's Encrypt

Traefik gestiona automáticamente los certificados SSL. La configuración relevante en .env:

VariableDesarrolloProducción
LETSENCRYPT_EMAILadmin@omnisalud.coadmin@omnisalud.co
LETSENCRYPT_CAhttps://acme-staging-v02...directoryvacío o comentado
COOKIE_SECUREfalsetrue
⚠️ En producción, elimine o comente LETSENCRYPT_CA para usar la CA real de Let's Encrypt. El valor staging solo emite certificados de prueba (no válidos en navegadores).
💡 Para desarrollo local sin HTTPS, no se requiere configuración de Let's Encrypt. Use http://localhost directamente.

5. Comandos de operación

5.1 Desplegar el ecosistema completo

cd /home/sa/infra
docker compose up -d --build

La primera ejecución descarga imágenes base, instala dependencias y compila los frontends React. Tiempo estimado: 3–5 minutos. Ejecuciones posteriores son más rápidas porque Docker aprovecha la caché de capas.

ℹ️ El flag -d ejecuta los contenedores en segundo plano. El flag --build fuerza la reconstrucción de las imágenes. Si solo quiere levantar contenedores ya construidos, omita --build.

5.2 Detener los servicios

docker compose down

Detiene y elimina todos los contenedores, pero conserva los volúmenes (certificados SSL, datos persistentes). Para eliminar también los volúmenes:

docker compose down -v

5.3 Reconstruir una aplicación específica

Cuando solo necesita actualizar una app sin afectar a las demás:

# Reconstruir solo app3 (Control de Costos)
docker compose up -d --build app3-costos

# Reconstruir el portal completo
docker compose up -d --build portal-auth portal-frontend

# Reconstruir app1
docker compose up -d --build app1-asistencial

# Reconstruir app2
docker compose up -d --build app2-comercial

5.4 Ver logs

# Logs de todos los servicios (streaming)
docker compose logs -f

# Logs de un servicio específico
docker compose logs -f app3-costos

# Últimas 50 líneas de Traefik
docker compose logs --tail 50 traefik

5.5 Estado de los contenedores

docker compose ps

Muestra el estado de cada contenedor: Up (activo), healthy (saludable), unhealthy (con problemas) o exited (detenido).

6. Monitoreo y health checks

6.1 Dashboard de Traefik

Traefik expone un panel de monitoreo en el puerto 8080 (solo accesible desde localhost):

http://localhost:8080

El dashboard muestra las rutas activas, servicios, middlewares y el estado de los certificados SSL.

6.2 Endpoints de salud

ServicioEndpointRespuesta esperada
portal-authGET /api/v1/health{"status":"healthy"}
app2-comercialGET /app2/health{"status":"healthy"}
# Verificar salud desde el host
curl -f http://localhost:80/api/v1/health
curl -f http://localhost:80/app2/health

6.3 Verificación post-despliegue

Ejecute estas comprobaciones después de cada despliegue:

# 1. Todos los contenedores en ejecución
docker compose ps

# 2. Endpoints responden
curl -f http://localhost:80/api/v1/health || echo "FAIL: auth"
curl -f -o /dev/null http://localhost:80/ || echo "FAIL: frontend"

# 3. Sin errores en logs recientes
docker compose logs --tail 30 | grep -i "error\|fail"

7. Ngrok — túnel para testing remoto

Ngrok permite exponer el ecosistema completo a internet sin configurar DNS, útil para pruebas con usuarios externos o demostraciones.

Opción A: Ngrok CLI (recomendado)

# 1. Asegurar que el stack está corriendo
docker compose up -d

# 2. Iniciar ngrok
ngrok http http://localhost:80

# 3. Usar la URL pública que muestra ngrok
#    Forwarding  https://xxxx-xx-xxx.ngrok-free.app -> http://localhost:80

Opción B: Ngrok en Docker

# 1. Configurar token en .env
#    NGROK_AUTHTOKEN=tu-token-aqui

# 2. Iniciar con perfil ngrok
docker compose --profile ngrok up -d

# 3. Obtener la URL pública
curl -s http://localhost:4040/api/tunnels | \
  python3 -c "import sys,json; print(json.load(sys.stdin)['tunnels'][0]['public_url'])"

8. Backup y restauración

8.1 Bases de datos

# Backup
mysqldump -h <host> -u <user> -p omn_core_global > backup_global_$(date +%Y%m%d).sql
mysqldump -h <host> -u <user> -p omn_asis_gestion_turnos > backup_asis_$(date +%Y%m%d).sql
mysqldump -h <host> -u <user> -p comercialDB > backup_comercial_$(date +%Y%m%d).sql
mysqldump -h <host> -u <user> -p omn_ng_control_costos_rn > backup_costos_$(date +%Y%m%d).sql

# Restauración
mysql -h <host> -u <user> -p omn_core_global < backup_global_20260101.sql

8.2 Archivos de configuración (backup mínimo)

# Respaldar configuración esencial
tar -czf infra_backup_$(date +%Y%m%d).tar.gz \
  /home/sa/infra/.env \
  /home/sa/infra/docker-compose.yml \
  /home/sa/infra/traefik/

8.3 Certificados Let's Encrypt

tar -czf letsencrypt_backup.tar.gz /home/sa/infra/traefik/letsencrypt/

9. Resolución de problemas

ProblemaCausa probableSolución
404 en todas las rutas Traefik no detecta los contenedores docker compose up -d --force-recreate
Error de conexión a MySQL Host inalcanzable o credenciales incorrectas Verificar MYSQL_HOST en .env; probar conectividad con mysql -h <host> -u <user> -p
Token JWT inválido entre apps JWT_SECRET_KEY no coincide Verificar que .env tenga el mismo JWT_SECRET_KEY para todos los servicios
Cookie SSO no se envía COOKIE_DOMAIN mal configurado Dejar COOKIE_DOMAIN vacío para mismo dominio; solo configurar para subdominios
Contenedor en estado unhealthy Health check fallando Revisar logs: docker logs <contenedor> --tail 50
Certificado SSL no se genera LetsEncrypt CA de staging o dominio sin DNS Verificar LETSENCRYPT_CA (vacío para prod); confirmar que el DNS resuelve al servidor
Puerto 80/443 ya en uso Otro proceso escuchando en esos puertos Detener el servicio冲突: sudo systemctl stop nginx/apache2; verificar con sudo lsof -i :80
Error de permisos en docker.sock Usuario sin acceso al socket de Docker sudo usermod -aG docker $USER && newgrp docker
💡 Para diagnóstico profundo, inspeccione un contenedor en ejecución: docker exec -it app3-costos sh y revise logs internos o conectividad.

Red interna

Todos los contenedores comparten la red bridge omni_net (subred 172.28.0.0/16). Solo Traefik expone los puertos 80 y 443 al host. La comunicación entre servicios se realiza por nombre de contenedor (ej. portal-auth:8000).

Logs y rotación

Todos los servicios usan rotación de logs automática: máximo 3 archivos de 10 MB cada uno (json-file driver). Los logs de acceso de Traefik se emiten en formato JSON a stdout para facilitar su ingestión en sistemas de monitoreo.
